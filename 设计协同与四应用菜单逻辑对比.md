# 设计协同 vs 四应用（采购/供应链/设备/能源）菜单逻辑对比

## 一、核心差异总结

| 对比项 | 设计协同 | 四应用（采购/供应链/设备/能源） |
|--------|----------|--------------------------------|
| 菜单来源 | main.html 静态 HTML | AppSidebarMenu.render() 动态渲染 |
| restoreAppState 是否调用 updateMenu | **否**（直接 return） | **是**（会重新渲染菜单） |
| 菜单是否会被重置 | 否 | **是**（见下） |

## 二、restoreAppState 行为对比

### 设计协同
```javascript
if (savedApp === 'design-collab') {
    console.log('ℹ️ 当前是默认应用，无需恢复UI');
    return;  // ← 直接返回，不调用 updateMenu
}
```
- 不会重新渲染菜单
- 菜单保持 main.html 中的静态结构
- loadPage 中的 setActiveForPage / updateActiveMenu 生效后不会被覆盖

### 四应用
```javascript
// 会执行到这里
updateMenu(config.menu);  // ← 调用 AppSidebarMenu.render()
```
- `AppSidebarMenu.render()` 会 `container.innerHTML = html`，完全替换菜单 DOM
- render 内部只调用 `expandDefault()`，仅展开「工作台」
- **不会**根据当前 URL 调用 `setActiveForPage`
- 结果：当前页面的激活状态和对应一级菜单的展开状态被清空

## 三、时序问题（根本原因）

页面加载流程：
1. DOMContentLoaded → PageLoader.loadPage(initialPage) 开始（异步）
2. DOMContentLoaded → setTimeout(restoreAppState, 100)

两种时序：
- **loadPage 先完成**：setActiveForPage 正确设置菜单 → 约 100ms 后 restoreAppState 执行 updateMenu → 菜单被重新渲染，状态被重置
- **restoreAppState 先完成**：updateMenu 重置菜单 → loadPage 完成后 setActiveForPage 再次设置 → 状态正确

因此问题表现为：本地/快速加载时，restoreAppState 在 loadPage 之后执行，导致菜单被错误重置。

## 四、二级菜单点击后的流程（相同）

设计协同和四应用在「点击二级菜单 → 加载页面」时的流程相同：
1. page-loader 的 document 点击处理器拦截链接
2. PageLoader.loadPage(url)
3. loadPage 内调用 AppSidebarMenu.setActiveForPage(url) 或 updateActiveMenu(url)
4. loadPage 封装的 wrapper 在 rAF 中再次调用 setActiveForPage

区别在于：四应用在 restoreAppState 时会被 updateMenu 覆盖一次（刷新或带 URL 初次加载时），而设计协同不会。

## 五、控制台多次「菜单已渲染」的原因

「菜单已渲染: procurement-menu」和「恢复应用状态: procurement」出现多次，可能原因：
1. restoreAppState 仅在首次执行时调用 updateMenu，不会重复执行（有 appStateInitialized  guards）
2. 若出现多次，更可能是：
   - 用户多次刷新
   - 或存在其他路径触发 switchApp / updateMenu（需再排查）

## 六、修复方案

在 restoreAppState 中，对四应用执行 updateMenu 之后，**必须根据当前 URL 调用 setActiveForPage**，恢复正确的激活项和展开状态。

```javascript
// 更新菜单
updateMenu(config.menu);

// 关键：根据当前 URL 恢复菜单激活/展开状态（与 loadPage 中逻辑一致）
var pageParam = new URLSearchParams(window.location.search).get('page');
if (pageParam && window.AppSidebarMenu && AppSidebarMenu.setActiveForPage) {
    AppSidebarMenu.setActiveForPage(pageParam);
} else if (pageParam && PageLoader && PageLoader.updateActiveMenu) {
    PageLoader.updateActiveMenu(pageParam);
}
```

这样无论 restoreAppState 与 loadPage 的执行先后如何，最终菜单状态都会正确。
